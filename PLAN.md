# Fix Composition Enrichment: fallback resilience, suggestion accuracy, and error visibility

## Context

The Composition Queue feature (`/composition-queue`) matches medicine names against a reference "salt master" list to auto-fill drug compositions, routing low-confidence matches to a manual review queue. Investigation (two rounds of codebase exploration, cross-checked directly against source) turned up three real, verified problems, and the user confirmed they want all three fixed:

1. **User-reported symptom**: "I didn't see the medicine list of the enrichment in page." Root cause: `CompositionQueue/index.tsx` swallows all fetch failures with only `console.error` — no error state exists. A failed `getEnrichmentStatus`/`getEnrichmentQueue` call renders the exact same "Run enrichment to start matching compositions" empty-state message as a genuinely empty queue, so the user can't tell a broken load from an empty one.
2. **User's explicit ask**: "add 2-3 fallback if one fails" for the online-enrichment step that runs when a medicine can't be matched locally. Today there are only two tiers (OpenFDA API, then a Google-search-via-OCR scrape), each tried exactly once with no retry — `OpenFdaClient.queryMedicine` catches every failure mode internally (timeout, HTTP error, empty result, 429) and returns `null` rather than throwing, so a naive "retry on exception" wrapper would never actually fire.
3. **Suggestion/confidence mismatch** (found during investigation, folds into "fix the enrichment"): `GET /enrichment/queue` re-derives the "suggested composition" shown for `needs_review` rows via a separate, much weaker `LIKE '%firstToken%'` query, run once per row (N+1), instead of using the actual reference row that produced the stored confidence score during matching. The number and the suggestion shown to the user can point at two different things.

(One earlier "bug" I reported — malformed backslash paths in `frontend/src/services/api.ts` — was checked directly and does not exist; discarded.)

## Fix 1 — Frontend error visibility (`frontend/src/pages/CompositionQueue/index.tsx`)

- Add two new state vars near the existing ones (~line 49-54): `statusError` and `queueError` (kept separate since the two fetches fail independently).
- In `loadStatus` ([index.tsx:58-66](frontend/src/pages/CompositionQueue/index.tsx#L58-L66)): `setStatusError(null)` on success, `setStatusError('Failed to load enrichment status.')` in the catch (alongside the existing `console.error`).
- In `loadQueue` ([index.tsx:68-81](frontend/src/pages/CompositionQueue/index.tsx#L68-L81)): same pattern with `queueError`.
- In the table body's conditional chain ([index.tsx:339-347](frontend/src/pages/CompositionQueue/index.tsx#L339-L347)), add a branch checked *before* the `queue.length === 0` empty-state: if `queueError` is set, render a distinct red/amber row (reuse the existing `AlertTriangle`/`XCircle` icons and the app's existing error color convention, e.g. `bg-red-500/10 text-red-400`) with the message and a "Retry" button calling `loadQueue()`.
- If `statusError` is set, show a small inline note near the dashboard header so a failed status poll isn't silently invisible either.

## Fix 2 — Retry + third fallback tier for online enrichment

**New file `src/utils/retry.ts`**: a small `withRetry<T>(fn: () => Promise<T | null>, opts?: { retries?: number; delayMs?: number; label?: string }): Promise<T | null>` helper. Because `OpenFdaClient.queryMedicine` (and the Google-search fallback) already swallow all internal errors and return `null` on any failure — never throw — this wrapper retries **on a `null` result**, not just on a thrown exception, up to `retries` attempts (default 2 retries = 3 total tries) with a short linear backoff (e.g. 400ms * attempt). This is a deliberate, simple call-site wrapper; no changes to the existing clients' internals or the `BaseApiClient` interface.

**Modify `src/services/onlineDataEnricher.ts`**: wrap each `client.queryMedicine(name)` call (in both `enrichMedicineData` and `enrichMedicineByName`, in the `for (const client of this.apiClients)` loops) and the Google-search fallback call (`googleSearchService.discoverMedicineInfo`) with `withRetry`. `OpenFdaClient`'s existing 429 circuit-breaker cooldown still applies underneath — a retry during an active cooldown returns `null` immediately (no extra network calls), so this doesn't risk hammering the API.

**New file `src/services/apiClients/rxNormClient.ts`**: `RxNormClient extends BaseApiClient`, `name = 'RxNorm'`, implementing `queryMedicine(name)` against the free, keyless NLM RxNorm REST API (`https://rxnav.nlm.nih.gov/REST/`): try `rxcui.json?name=<name>` for an exact RXCUI match, fall back to `approximateTerm.json?term=<name>` (take the top-scored candidate) if no exact match, then call `related.json?tty=IN+PIN+MIN` on the resolved RXCUI to pull ingredient names into `activeIngredients`. Match `OpenFdaClient`'s conventions: 5s `AbortSignal.timeout`, catch-and-return-null on any failure, `source: this.name`. Other `EnrichedProductData` fields (`indications`, `dosage`, etc.) are legitimately left `undefined` — RxNorm is a terminology/ingredient service, not a label database, but `activeIngredients` is the only field this feature actually consumes (joined into the `api_reference` composition string).

**Modify `src/services/onlineDataEnricher.ts` constructor** (currently `this.apiClients.push(new OpenFdaClient())`): also push `new RxNormClient()`, positioned after OpenFDA. The existing generic `for (const client of this.apiClients)` loop picks it up automatically — no loop changes needed. Resulting chain: OpenFDA (retried) → RxNorm (retried, new) → Google-search scrape (retried) → give up.

## Fix 3 — Suggested-composition accuracy (remove the mismatch + N+1)

**Modify `src/worker/compositionEnricher.ts`**: add one more lazy column via the same existing pattern already used for `enrichment_status`/`enrichment_confidence` ([lines 192-201](src/worker/compositionEnricher.ts#L192-L201)): `ALTER TABLE medicines ADD COLUMN suggested_composition TEXT DEFAULT NULL`, wrapped in the same try/catch-ignore-if-exists idiom. In the `needs_review` branch ([lines 323-334](src/worker/compositionEnricher.ts#L323-L334)), where `composition` is already computed from the exact `bestRef` that produced the confidence score, add `suggested_composition = ?` to the existing `UPDATE` statement using that same variable — guaranteeing the stored suggestion and stored confidence always come from the identical matched row. While here, delete the now-truly-dead `calculateMatchScore` function ([lines 44-69](src/worker/compositionEnricher.ts#L44-L69)) since it's unused and diverges from the inline scoring logic that's actually used.

**Modify `src/routes/enrichment.ts`**: in `GET /enrichment/queue` ([lines 144-191](src/routes/enrichment.ts#L144-L191)), delete the N+1 `LIKE` loop ([lines 165-177](src/routes/enrichment.ts#L165-L177)) entirely and add `suggested_composition` directly to the main `SELECT` ([lines 160-163](src/routes/enrichment.ts#L160-L163)). One query, no follow-ups, and the suggestion is now structurally guaranteed to match the confidence shown next to it. Frontend needs no changes here — `QueueItem.suggested_composition` ([CompositionQueue/index.tsx:22](frontend/src/pages/CompositionQueue/index.tsx#L22)) already expects exactly this field.

**Backfill note**: existing rows already marked `needs_review` from a prior run won't have this new column populated (since `runEnrichment` only ever processes rows where `enrichment_status IS NULL`, and won't touch already-categorized rows). This is left as-is — no destructive reset of existing review progress. The frontend already renders gracefully when `suggested_composition` is absent (falls through to the plain manual-entry input), so old rows just show the manual-entry box until re-enrichment tooling exists; nothing breaks.
